# PetCare 宠伴 - 架构基线与演进决策

**最后更新**：2026-07-21
**维护者**：PetCare 开发团队
**状态**：已采纳

## 1. 决策摘要

PetCare 在 MVP 及早期增长阶段采用 **模块化单体 + 异步 Worker**：一个 NestJS API 应用按业务领域划分模块，一个或多个独立 Worker 处理队列任务；PostgreSQL 是业务事务的唯一主数据源，Redis 用于缓存、限流和 BullMQ 队列。

不在 MVP 阶段按技术层或预想的服务名拆分微服务，也不同时维护 MySQL、PostgreSQL 和 MongoDB。服务拆分必须由可观测的负载、合规边界或团队自治需求触发。

## 2. 当前目标架构

```text
小程序 / Admin
      │ HTTPS
      ▼
NestJS API（模块化单体） ────────────────► PostgreSQL（事务主库）
      │                                      │
      ├── Redis（缓存、限流、BullMQ）          └── Outbox 事件表
      │                                      │
      └──────────────────────────────────────┘
                                             ▼
                                  BullMQ Worker（通知、结算、超时任务）
                                             │
                      微信支付 / 订阅消息 / 对象存储 / LBS 等外部服务
```

API 和 Worker 可以独立扩容；它们共享同一版本的领域模型、队列前缀和环境配置。Worker 不直接承担同步 HTTP 请求。

## 3. 模块边界

模块按业务能力组织，模块间只能通过公开的应用服务接口或领域事件协作，禁止跨模块直接访问持久化实现。

| 领域       | 责任                                 | 后续可独立的条件                       |
| ---------- | ------------------------------------ | -------------------------------------- |
| 身份与权限 | 登录、用户、服务者认证、RBAC、审计   | 身份合规或多端 SSO 需要独立边界        |
| 订单履约   | 悬赏/平台订单、派单、SOP、评价、投诉 | 订单写入量或履约流程成为主要瓶颈       |
| 支付结算   | 支付回调、退款、账本、提现           | 支付合规、对账复杂度或独立发布频率提高 |
| 通知       | 站内信、订阅消息、推送               | 队列吞吐或供应商集成需要单独扩缩容     |
| 内容社区   | 帖子、互动、审核、搜索               | 读流量显著高于交易流量，或审核流程独立 |
| 运营分析   | 聚合指标、报表导出                   | 分析查询影响交易库或需要数仓能力       |

新增模块放在 `apps/server/src/modules/<domain>`；跨领域的队列、文件、监控等能力放在 `apps/server/src/infrastructure`，不能反向依赖业务模块。

## 4. 数据与一致性规则

1. PostgreSQL 是订单、支付、结算、用户及权限数据的权威来源；Redis 只保存可重建的缓存和队列状态。
2. 所有金额使用 PostgreSQL `numeric`（Prisma `Decimal`）或最小货币单位整数，禁止使用 `Float`。
3. 订单状态、支付状态和库存/名额类资源使用数据库事务与条件更新，接口提供幂等键。
4. 在同一数据库事务内写入业务变更和 Outbox 事件；Worker 读取事件并投递 BullMQ。消费者按事件 ID 去重，允许至少一次投递。
5. 外部回调（尤其支付）必须验签、幂等，并保留审计记录；不得以回调到达顺序替代状态机校验。

## 5. 异步任务约定

首批队列包括：订单超时关闭、通知发送、图片/视频处理、结算和运营聚合。每个任务都必须声明：任务名、输入 schema、幂等键、重试策略、最大执行时间和失败告警。

以下变量由 `ConfigService` 统一读取，生产者与 Worker 必须保持一致：

| 变量                      | 默认值      | 用途                        |
| ------------------------- | ----------- | --------------------------- |
| `QUEUE_PREFIX`            | `petcare`   | 隔离不同环境的 BullMQ 队列  |
| `WORKER_CONCURRENCY`      | `5`         | 单 Worker 的并发任务数      |
| `OUTBOX_POLL_INTERVAL_MS` | `1000`      | Outbox 轮询周期（毫秒）     |
| `ORDER_TIMEOUT_DELAY_MS`  | `172800000` | 悬赏订单关闭延迟（48 小时） |

## 6. 拆分门槛

满足以下任一条件前，不拆分服务：

- 某领域需要独立的合规、密钥、审计或故障隔离边界；
- 该领域的 CPU、内存、队列吞吐或发布频率持续拖累其他领域；
- 有稳定的事件契约、可独立监控的 SLO，以及明确负责该服务的团队；
- 压测或生产指标确认通过 API/Worker 水平扩展、PostgreSQL 索引/分区或只读副本仍无法满足目标。

首次优先候选是支付结算和通知，其次是内容社区。拆分时先复制事件到新消费者，完成双写/回放验证后再迁移流量；禁止把共享数据库作为长期服务间接口。

## 7. 部署演进

| 阶段   | 部署                                          | 数据策略                                   |
| ------ | --------------------------------------------- | ------------------------------------------ |
| MVP    | Docker Compose：Admin、API、PostgreSQL、Redis | 单 PostgreSQL 主库，定期备份               |
| 增长期 | 多副本 API + 独立 Worker                      | 连接池、索引、慢查询治理、只读副本按需引入 |
| 服务化 | Kubernetes 或等价编排平台                     | 按领域数据所有权迁移，事件契约版本化       |
