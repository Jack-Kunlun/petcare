# 管理后台图形验证码设计

## 背景与目标

管理后台目前支持手机号发送短信验证码。为降低自动化脚本批量触发短信、消耗短信额度和探测账号的风险，在发送短信验证码之前增加服务端图形验证码校验。

本次只保护“发送短信验证码”操作，不改变密码登录、短信验证码登录、Token 刷新与退出登录流程。

## 方案选择

采用后端生成 SVG、Redis 保存摘要并原子单次消费的自建方案。

未选择第三方滑块验证码，因为当前本地开发阶段不需要额外供应商、费用与回调集成。未选择纯前端算术验证码，因为答案可由脚本直接读取，不能形成有效安全边界。

## 接口设计

### 获取图形验证码

`GET /auth/captcha`

响应：

```json
{
  "captchaId": "随机不可预测的标识",
  "image": "data:image/svg+xml;base64,...",
  "expiresIn": 300
}
```

行为：

- 使用密码学安全随机源生成验证码标识和 4 位验证码。
- 字符集仅包含数字 `2–9`，避免 `0/1` 等易混淆字符。
- 数字使用服务端内置的七段矢量路径绘制，SVG 中禁止出现 `<text>` 节点或可直接解析出答案的字符属性。
- SVG 包含字形偏移、旋转、干扰线和噪点，但保证正常人工可读。
- SVG 内容完全由服务端受控字符生成，不接受用户输入。
- 响应不返回验证码答案，不设置用户或管理员相关状态。

### 发送短信验证码

`POST /auth/sms/send`

请求：

```json
{
  "phone": "17679141878",
  "captchaId": "获取图形验证码时返回的标识",
  "captchaCode": "用户看到并输入的4位验证码"
}
```

行为顺序：

1. DTO 校验手机号、验证码标识和 4 位验证码格式。
2. 原子校验并消费图形验证码。
3. 图形验证码无效时返回统一的 `400` 错误，不查询管理员、不发送短信。
4. 图形验证码有效时，执行现有管理员检查、短信发送冷却和每小时发送上限。
5. 无论手机号是否对应管理员，接口继续使用统一成功消息，避免账号枚举。

图形验证码校验成功后立即失效。短信发送失败时也不恢复，用户必须获取新图形验证码后重试。

## 后端组件

### CaptchaService

职责：

- 生成验证码标识、答案和 SVG 图片。
- 使用 `JWT_SECRET` 作为 HMAC 密钥，对 `captchaId` 与规范化答案计算摘要。
- 将摘要写入 Redis，不保存可直接读取的答案明文。
- 调用 Redis 原子操作完成尝试次数判断、校验和单次消费。

Redis 键：

- `auth:captcha:<captchaId>`：验证码摘要。
- `auth:captcha:attempts:<captchaId>`：失败次数。

默认策略：

- 有效期 300 秒。
- 最多失败 5 次。
- 输入统一移除首尾空白后校验。
- 成功后立即删除摘要和失败次数。
- 超过失败上限后删除验证码并拒绝后续校验。

### RedisService

新增通用的原子“校验并消费一次性摘要”操作，复用当前短信 OTP Lua 脚本的安全模型。比较、失败计数和删除必须在单个 Redis 脚本中完成，避免并发请求重复消费同一验证码。

### AuthService 与 DTO

`SendSmsCodeDto` 增加 `captchaId`、`captchaCode`。`AuthService.sendSmsCode` 在任何用户查询之前调用 `CaptchaService.verifyAndConsume`。

## 前端交互

验证码登录标签页增加：

- 图形验证码输入框。
- 图形验证码图片。
- 点击图片刷新能力和明确的“换一张”无障碍名称。

数据流：

1. 用户首次切换到验证码登录标签页时请求图形验证码。
2. 用户输入手机号和图形验证码后点击“发送验证码”。
3. 前端提交 `phone`、`captchaId` 和 `captchaCode`。
4. 成功后启动现有 60 秒倒计时、清空图形验证码输入并获取新图片。
5. 图形验证码错误、过期或短信发送失败时，展示安全错误、清空输入并获取新图片。
6. 图片加载失败时显示重试按钮，不允许在缺少有效 `captchaId` 时发送短信。

前端不在 Local Storage 或 Session Storage 持久化验证码标识和答案。

## 错误处理

- 缺少或格式错误：DTO 返回 `400`。
- 答案错误、已过期、已消费或超过尝试次数：统一返回 `400` 和“图形验证码错误或已过期”。
- Redis 不可用：明确返回服务错误，不降级为进程内存验证码。
- 短信发送过于频繁：保留现有 `429` 行为。
- 前端对所有失败都刷新图形验证码，避免重复提交失效挑战。

## 配置

新增配置：

```bash
CAPTCHA_TTL_SECONDS=300
CAPTCHA_MAX_ATTEMPTS=5
```

配置通过后端 `ConfigService` 读取并验证为正整数，同时同步 `.env.example` 与环境变量文档。

## 测试策略

后端单元测试：

- 生成的验证码标识不可预测，响应包含 SVG Data URL，但不包含答案字段、`<text>` 节点或可直接解析的答案字符。
- Redis 只保存摘要，不保存答案明文。
- 大小写规范化后可成功校验。
- 正确答案只能消费一次。
- 错误答案增加尝试次数，达到上限后失效。
- 过期或不存在的验证码失败。
- 未通过图形验证码时不查询管理员、不发送短信。
- 图形验证码通过后才进入原有短信发送流程。
- 控制器和 DTO 正确传递三个请求字段。

Admin 单元测试：

- 切换到验证码登录时加载图片。
- 缺少手机号、图形验证码或挑战标识时不调用发送接口。
- 发送请求包含手机号、验证码标识和用户答案。
- 点击图片可换一张。
- 成功和失败后均清空输入并刷新图片。
- 图片加载失败时可手动重试。

最终验证包括 Server/Admin 全量测试、Lint、构建、Prisma 校验、格式检查，以及本地 PostgreSQL/Redis 下的真实 HTTP 图形验证码与短信发送联调。

## 验收标准

- 没有有效图形验证码时无法触发短信发送逻辑。
- 同一个图形验证码不能重复发送短信。
- 验证码答案不会通过 API、日志或 Redis 明文泄露。
- 现有密码登录、短信登录、会话恢复和退出登录行为不回归。
- 本地 Admin 页面可以完成“图形验证码 → 发送短信验证码 → 短信登录”的完整流程。
