# PetCare 统一分页响应设计

**日期**：2026-07-23  
**状态**：已确认，待实施

## 1. 背景

当前项目存在两种分页列表字段：共享分页类型使用 `items`，订单列表接口使用 `orders`。字段名随领域变化会增加客户端通用分页组件、API Client 和 Swagger 文档的适配成本。

本次变更将所有分页接口的业务数据统一为同一结构，并继续放在统一 API 响应的 `data` 字段中。

## 2. 统一契约

所有 JSON 分页接口必须返回以下结构：

```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": {
    "list": [],
    "total": 0,
    "page": 1,
    "pageSize": 20
  },
  "meta": {
    "requestId": "request-id",
    "timestamp": "2026-07-23T00:00:00.000Z"
  }
}
```

字段约束：

- `list`：当前页数据，始终为数组；没有数据时返回空数组。
- `total`：符合查询条件的记录总数，为非负整数。
- `page`：当前页码，从 1 开始。
- `pageSize`：每页记录数，为正整数。

分页字段属于业务数据，因此位于 `data` 内；`meta` 继续只承载请求追踪和响应时间信息。

## 3. 实现边界

- 共享类型 `PaginatedResponse<T>` 使用 `list: T[]`，作为唯一通用分页类型。
- 订单列表响应类型、服务实现和 Swagger DTO 使用 `list`。
- 服务层直接产生统一分页对象，Controller 不执行字段重命名。
- API Client 和 Admin 继续只负责统一响应外层解包，不对分页字段做隐式转换。
- API 规范、架构示例和 Swagger 输出同步采用 `list`。

## 4. 兼容策略

项目仍处于早期开发阶段，本次直接切换，不保留 `items`、`orders` 等旧列表字段，也不引入双写或兼容转换层。后续新增分页接口必须直接复用统一分页类型。

## 5. 测试策略

按测试先行方式实施：

1. 共享类型契约测试验证分页帮助对象只包含 `list`、`total`、`page`、`pageSize`。
2. 订单服务测试先断言返回 `list` 且不存在 `orders`，确认旧实现失败后再修改生产代码。
3. Swagger 测试验证订单列表响应的 `data.list` 是订单 DTO 数组。
4. 执行相关单元测试后，再运行全量测试、Lint、构建和格式检查。

## 6. 完成标准

- 仓库内运行时代码和公开类型不再使用 `items` 或领域化列表字段表示分页数据。
- `GET /orders` 的 `data` 精确包含 `list`、`total`、`page`、`pageSize`。
- Swagger 与 API 文档展示相同结构。
- 相关测试经历预期失败和通过阶段，全量校验通过。
