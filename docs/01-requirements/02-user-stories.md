# PetCare宠伴 - 用户故事与验收标准（v2.2）

## 一、用户角色定义

### 1.1 核心角色

| 角色       | 英文标识         | 描述                                           |
| ---------- | ---------------- | ---------------------------------------------- |
| 宠物主人   | Pet Owner        | 需要上门喂养服务的宠物主，平台付费用户         |
| 服务提供者 | Service Provider | 提供上门服务的用户，包括认证宠托师和实名宠物主 |
| 平台管理员 | Admin            | 负责用户审核、订单管理、内容审核等后台运营     |

### 1.2 次要角色

| 角色     | 英文标识         | 描述                                     |
| -------- | ---------------- | ---------------------------------------- |
| 访客     | Visitor          | 未注册用户，可浏览社区内容和了解服务     |
| 社区用户 | Community Member | 注册但未使用服务的用户，主要参与社区互动 |

**重要说明**：

- **服务提供者分为两类**：
  1. **认证宠托师**：完成平台认证（身份认证+技能认证+微信支付分≥650），可接平台定价订单
  2. **实名宠物主**：完成实名认证，可在悬赏大厅接其他宠物主发布的悬赏订单
- **隐私保护原则**：宠物主端在接单前完全不展示服务提供者信息，只有接单后才建立联系

## 五、RBAC权限管理体系（新增）

### Epic 11：基于角色的访问控制

#### Story 11.1：角色管理

**作为** 超级管理员  
**我希望** 能够创建和管理系统角色  
**以便** 为不同职责的用户分配合适的权限

**验收标准**：

- [ ] 支持创建自定义角色（如：客服专员、运营经理、财务审核员、商家管理员等）
- [ ] 每个角色可配置名称、描述、所属模块
- [ ] 预设系统角色：超级管理员、客服主管、普通客服、运营专员、财务专员、商家管理员
- [ ] 角色可启用/禁用，禁用后该角色下所有用户权限失效
- [ ] 角色删除前需检查是否有用户关联，有关联则禁止删除
- [ ] 支持角色复制，快速创建相似角色

---

#### Story 11.2：权限点管理（按钮级别）

**作为** 超级管理员  
**我希望** 能够定义细粒度的权限点  
**以便** 实现按钮级别的权限控制

**验收标准**：

- [ ] 权限点按模块组织：用户管理、订单管理、内容审核、财务管理、数据统计、系统设置、商家管理等
- [ ] 每个权限点包含：权限编码（唯一）、权限名称、所属模块、权限类型（菜单/按钮/API）
- [ ] 按钮级别权限示例：
  - **用户管理**：查看用户列表、编辑用户信息、冻结用户、重置密码
  - **订单管理**：查看订单列表、查看订单详情、手动取消订单、强制退款、导出订单
  - **纠纷处理**：查看投诉列表、处理投诉、发起二次裁决、查看裁决历史
  - **内容审核**：查看待审核内容、通过审核、拒绝审核、批量操作
  - **财务管理**：查看收入明细、审核提现、手动结算、导出财务报表
  - **商家管理**：创建商家账号、编辑商家信息、查看商家数据、冻结商家
- [ ] 权限点支持批量导入/导出（Excel格式）
- [ ] 新增权限点后需重新分配给相关角色才生效

---

#### Story 11.3：角色权限分配

**作为** 超级管理员  
**我希望** 能够为角色分配权限  
**以便** 控制不同角色的操作范围

**验收标准**：

- [ ] 可视化权限树展示，支持勾选/取消权限
- [ ] 支持全选/全不选某个模块下的所有权限
- [ ] 权限分配实时保存，无需额外确认
- [ ] 显示当前角色已分配的权限数量
- [ ] 支持权限对比功能，查看两个角色的权限差异
- [ ] 权限变更后，已登录用户下次刷新页面时生效

---

#### Story 11.4：用户角色分配

**作为** 超级管理员  
**我希望** 能够为用户分配一个或多个角色  
**以便** 灵活控制用户的权限组合

**验收标准**：

- [ ] 支持为用户分配多个角色，权限取并集
- [ ] 分配角色时显示该角色的权限概要
- [ ] 支持批量为用户分配角色
- [ ] 移除角色时需二次确认
- [ ] 用户至少拥有一个角色，禁止移除最后一个角色
- [ ] 显示用户的完整权限列表（合并所有角色的权限）

---

#### Story 11.5：前端按钮级权限控制

**作为** 系统开发人员  
**我希望** 前端能够根据用户权限动态显示/隐藏按钮  
**以便** 实现真正的按钮级别权限控制

**验收标准**：

- [ ] 前端加载时获取当前用户的权限列表
- [ ] 使用权限指令/组件控制按钮显示：`v-permission="'user:edit'"`
- [ ] 无权限的按钮完全隐藏（不是禁用），避免用户困惑
- [ ] API接口层也需做权限校验，防止绕过前端直接调用
- [ ] 权限变更后可通过刷新页面或重新登录生效
- [ ] 提供权限调试模式，开发环境可查看当前用户所有权限

---

#### Story 11.6：商家端权限隔离（面向未来扩展）

**作为** 系统设计者  
**我希望** RBAC体系能够支持未来商家端的扩展  
**以便** 无需重新设计权限体系即可服务商家

**验收标准**：

- [ ] 权限体系中预留"商家管理"模块
- [ ] 商家相关权限点包括：
  - **商家入驻**：审核商家申请、查看商家资料、批准/拒绝入驻
  - **商家订单**：查看商家订单、处理商家订单纠纷、导出商家订单
  - **商家财务**：查看商家收入、审核商家提现、生成商家对账单
  - **商家数据**：查看商家经营数据、生成商家报表、导出数据
  - **商家设置**：编辑商家信息、配置商家费率、冻结/解冻商家
- [ ] 商家管理员角色只能查看和操作自己商家的数据（数据权限隔离）
- [ ] 平台管理员可以查看所有商家数据
- [ ] 权限编码规范统一，便于后续扩展（如：`merchant:view`, `merchant:order:manage`）
- [ ] 数据权限与功能权限分离，支持按商家ID过滤数据

---

#### Story 11.7：权限审计日志

**作为** 超级管理员  
**我希望** 能够查看权限变更的历史记录  
**以便** 追溯权限调整的责任人和时间

**验收标准**：

- [ ] 记录所有角色创建/修改/删除操作
- [ ] 记录所有权限分配/移除操作
- [ ] 记录操作用户、操作时间、操作IP、变更内容
- [ ] 支持按时间范围、操作人、操作类型筛选
- [ ] 审计日志不可删除，只可归档
- [ ] 敏感操作（如赋予超级管理员权限）需额外审批流程

## 八、非功能性需求

### 8.1 性能要求

- 首页加载时间 < 2秒
- 订单创建响应时间 < 1秒
- 图片上传压缩后单张 < 2MB
- 支持并发用户数 ≥ 10000

### 8.2 安全要求

- 用户敏感信息（身份证、手机号）加密存储
- 支付接口采用HTTPS加密传输
- 防止SQL注入和XSS攻击
- 定期备份数据库（每日全量备份）
- 实名认证数据对接公安系统验证
- **RBAC权限校验**：所有API接口需进行权限验证

### 8.3 兼容性要求

- 支持iOS 12+ 和 Android 8.0+
- 适配主流屏幕尺寸（375x667 ~ 414x896）
- 微信小程序基础库版本 ≥ 2.10.0

### 8.4 可用性要求

- 系统可用性 ≥ 99.9%
- 支持7×24小时服务
- 故障恢复时间 < 30分钟

| 优先级 | 说明        | 包含Epic                                                                                 |
| ------ | ----------- | ---------------------------------------------------------------------------------------- |
| P0     | MVP必须实现 | Epic 1, 2, 5(Story 5.1), 6, 7(Story 7.1-7.2)                                             |
